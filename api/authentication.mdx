Every request to the Geo Web IW REST API must include a valid API key. Pass your key as a Bearer token in the `Authorization` header — no cookies, no sessions, no OAuth flow required. Keys are tied to your account and carry the specific scopes you assign at creation time.

<Tip>
  Follow the principle of least privilege: create a separate key for each integration and grant only the scopes that integration actually needs. This limits the blast radius if a key is ever compromised.
</Tip>

## Generating an API Key

<Steps>
  <Step title="Log in to your account">
    Navigate to [app.geowebhq.com](https://app.geowebhq.com) and sign in with your credentials.
  </Step>
  <Step title="Open Settings">
    Click your avatar in the top-right corner and select **Settings** from the dropdown menu.
  </Step>
  <Step title="Go to API Keys">
    In the left sidebar, select **API Keys** under the **Developer** section.
  </Step>
  <Step title="Create a new key">
    Click **New Key**, choose the scopes your integration requires, and click **Generate**.
  </Step>
  <Step title="Name your key">
    Give the key a descriptive name — for example, `prod-data-pipeline` or `staging-dashboard` — so you can identify and rotate it later.
  </Step>
  <Step title="Copy and store the key">
    Copy the key immediately and store it in a secrets manager or environment variable.
  </Step>
</Steps>

<Warning>
  Your API key is displayed **only once** at creation time. If you navigate away without copying it, you must delete the key and generate a new one. Never commit keys to source control.
</Warning>

## Making Authenticated Requests

Pass your API key in the `Authorization` header on every request using the `Bearer` scheme.

```
Authorization: Bearer YOUR_API_KEY
```

All Geo Web IW API keys begin with the prefix `gwk_` so you can identify them at a glance in logs and configuration files.

<CodeGroup>
  ```bash cURL
  curl https://api.geowebhq.com/v1/maps \
    -H "Authorization: Bearer gwk_your_api_key_here"
  ```

  ```javascript JavaScript (fetch)
  const response = await fetch('https://api.geowebhq.com/v1/maps', {
    headers: {
      'Authorization': 'Bearer gwk_your_api_key_here',
      'Content-Type': 'application/json'
    }
  });
  const { data } = await response.json();
  ```
</CodeGroup>

<Note>
  Store your key in an environment variable such as `GEO_WEB_API_KEY` and reference it at runtime. Hardcoding keys in source files is a common cause of credential leaks.
</Note>

## API Key Scopes

When you create a key, select the scopes that match the operations your integration needs to perform. A key without a required scope will receive a `403 Forbidden` response for that operation even if the key itself is valid.

<Accordion title="Available scopes">
  | Scope | Permitted operations |
  |---|---|
  | `maps:read` | List and retrieve maps. |
  | `maps:write` | Create, update, and delete maps. |
  | `layers:read` | List and retrieve layers. |
  | `layers:write` | Create, update, and delete layers. |
  | `data:read` | Read data sources and query spatial data. |
  | `data:write` | Create, update, and delete data sources. |
  | `workflows:read` | List and inspect workflow definitions and run history. |
  | `workflows:write` | Create, trigger, update, and delete workflows. |
  | `admin` | Full access to all resources and account settings. Reserve this for trusted internal tooling only. |
</Accordion>

## Authentication Errors

The API returns one of two status codes when a request fails for authentication or authorization reasons.

<Tabs>
  <Tab title="401 Unauthorized">
    A `401 Unauthorized` response means the API could not authenticate your request. Common causes:

    - The `Authorization` header is missing from the request.
    - The header is present but malformed — for example, `Bearer` is misspelled or the token is truncated.
    - The key has been revoked or deleted from the dashboard.

    **Fix:** Verify that the header is formatted as `Authorization: Bearer gwk_...`, confirm the key is active in **Settings → API Keys**, and check that your environment variable is being injected correctly.
  </Tab>
  <Tab title="403 Forbidden">
    A `403 Forbidden` response means the key is valid but lacks the scope needed for the requested operation.

    - You are calling a write endpoint (`POST`, `PUT`, `DELETE`) but the key only has a `read` scope.
    - You are accessing an admin-only endpoint without the `admin` scope.

    **Fix:** Delete the current key and create a new one with the required scopes. You cannot add scopes to an existing key.
  </Tab>
</Tabs>

## Rotating API Keys

Rotate keys periodically — or immediately if you suspect a key has been exposed.

<Steps>
  <Step title="Create the replacement key">
    Generate a new key in **Settings → API Keys** with the same scopes as the key you are replacing. Test it in a non-production environment first.
  </Step>
  <Step title="Update all services">
    Replace the old key value in every service, CI/CD pipeline, and environment that uses it. Deploy and verify each service is operating correctly with the new key.
  </Step>
  <Step title="Delete the old key">
    Once all services are confirmed healthy, return to **Settings → API Keys** and delete the old key. Deleting it immediately invalidates it.
  </Step>
</Steps>

<Warning>
  Delete the old key **only after** all dependent services have been updated and verified. Deleting it prematurely will cause `401` errors in any service still using the old value.
</Warning>
